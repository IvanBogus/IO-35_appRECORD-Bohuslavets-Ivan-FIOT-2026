## Тема, мета, місце розташування

**Тема:** Реалізація серверної частини інформаційної системи для обліку студентів і навчальних груп із використанням бази даних MySQL, бібліотеки `mysql2` та `Sequelize ORM`.

**Мета:** дослідити структуру та основні компоненти серверної частини проєкту `lab1-rest-api`, що працює з базою даних `studentlab_db`, і узагальнити фактично реалізовані можливості роботи з даними через SQL-запити, `mysql2` та `Sequelize ORM`.

**Завдання:**
1. Проаналізувати структуру серверної частини проєкту.
2. Описати структуру бази даних `studentlab_db`.
3. Охарактеризувати SQL-скрипти, використані для створення схеми та демонстраційних запитів.
4. Визначити реалізовані маршрути API для роботи з даними через `mysql2`.
5. Визначити реалізовані маршрути API для роботи з даними через `Sequelize ORM`.
6. Описати реалізацію зв’язку типу «один-до-багатьох» між сутностями `Group` і `Student`.
7. Узагальнити результати тестування CRUD-операцій серверної частини.

**Короткий опис системи:** серверна частина реалізована на Node.js та Express.js у межах проєкту `lab1-rest-api`. Для лабораторної роботи 2 використовується база даних `studentlab_db`, яка містить таблиці `groups` і `students`. У проєкті реалізовано два підходи до роботи з базою даних: через сирі SQL-запити за допомогою `mysql2` та через `Sequelize ORM`. Для таблиць `groups` і `students` реалізовано зв’язок типу «один-до-багатьох», що дозволяє отримувати дані студентів разом із пов’язаними даними груп.

## 1. Реалізація бази даних та серверної частини

### 1.1 Загальний опис

Серверна частина лабораторної роботи реалізована на платформі Node.js із використанням фреймворку Express.js. Її призначення полягає у наданні REST API для роботи з даними студентів і навчальних груп, що зберігаються у базі даних MySQL. У проєкті передбачено два окремі підходи до доступу до даних: через SQL-запити за допомогою `mysql2` та через ORM `Sequelize`. Така реалізація дозволяє порівняти роботу з базою даних на рівні запитів і на рівні моделей.

### 1.2 Структура бази даних

Для лабораторної роботи 2 у проєкті використовується база даних `studentlab_db`. У її складі створено дві таблиці: `groups` і `students`. Таблиця `groups` містить основні поля `id`, `name`, `code`, `curator_name` та `study_year`, а таблиця `students` — поля `id`, `first_name`, `last_name`, `email`, `birth_date`, `enrollment_year` і `group_id`. Зв’язок між таблицями реалізовано через зовнішній ключ `students.group_id`, який посилається на `groups.id`. Така структура відповідає зв’язку типу «один-до-багатьох», за якого одна група може містити багато студентів, а кожен студент належить до однієї групи.

### 1.3 SQL-скрипти

У каталозі `sql/` розміщено два основні файли: `schema.sql` і `demo-queries.sql`. Файл `schema.sql` використовується для створення бази даних `studentlab_db`, таблиць `groups` і `students`, зовнішнього ключа між ними та початкових тестових записів. Файл `demo-queries.sql` містить приклади базових SQL-запитів `SELECT`, `INSERT`, `UPDATE` та `DELETE`. Такий поділ дає змогу окремо зберігати схему бази даних і приклади роботи з нею.

### 1.4 Підключення через mysql2

Для роботи з базою даних через SQL-запити у проєкті використовується бібліотека `mysql2`. Підключення налаштовано у файлі `config/database.js`, де створюється пул з’єднань на основі параметрів середовища. У маршрутах `routes/mysql2/groups.js` і `routes/mysql2/students.js` реалізовано отримання списку груп, отримання списку студентів разом із даними групи, а також додавання, оновлення та видалення записів студентів. Для студентів використовуються SQL-запити з перевіркою наявності коректного `group_id` перед створенням або оновленням запису.

### 1.5 Використання Sequelize ORM

Для другого підходу до роботи з базою даних у проєкті використовується `Sequelize ORM`. Підключення до MySQL налаштовано у файлі `config/sequelize.js`, а моделі описано у файлах `models/group.js` і `models/student.js`. У маршрутах `routes/sequelize/groups.js` і `routes/sequelize/students.js` реалізовано отримання списку груп, отримання списку студентів, а також додавання, оновлення та видалення студентів через ORM-моделі. Такий підхід дозволяє працювати з даними через об’єкти та методи моделей без ручного написання SQL-запитів у кожному маршруті.

### 1.6 Реалізація зв’язку One-to-Many

Зв’язок між сутностями `Group` і `Student` реалізовано у файлі `models/index.js`. Для цього використано зв’язки `Group.hasMany(Student, { foreignKey: "group_id" })` та `Student.belongsTo(Group, { foreignKey: "group_id" })`. У маршрутах `mysql2` цей зв’язок демонструється через `JOIN` між таблицями `students` і `groups`, а в маршрутах `Sequelize` — через використання `include` під час отримання студентів. Завдяки цьому сервер повертає не лише дані студента, а й пов’язану інформацію про його навчальну групу.

### 1.7 Тестування API

Під час тестування було перевірено маршрути `mysql2` для отримання списку груп і студентів, а також для додавання, оновлення та видалення записів студентів. Окремо було перевірено маршрути `sequelize` для отримання списку груп і студентів та виконання CRUD-операцій над записами студентів. У результаті тестування підтверджено коректне виконання операцій `GET`, `POST`, `PUT` і `DELETE` для обох варіантів реалізації доступу до бази даних. Під час спроби створити або оновити студента з неіснуючим значенням `group_id` сервер повертав повідомлення про помилку, що підтверджує перевірку зв’язку з таблицею груп. Зв’язок типу «один-до-багатьох» між `Group` і `Student` було продемонстровано під час отримання списку студентів разом із даними групи через `JOIN` у маршрутах `mysql2` та через `include` у маршрутах `Sequelize`.

## 2. Файлова структура проєкту

Структура серверної частини поділена на окремі каталоги конфігурації, моделей, маршрутів і SQL-скриптів.

### 2.1 Основна структура

Каталог `lab1-rest-api/` містить серверну частину застосунку. Основні елементи структури:

```text
lab1-rest-api/
├── config/
│   ├── database.js
│   └── sequelize.js
├── models/
│   ├── group.js
│   ├── index.js
│   └── student.js
├── routes/
│   ├── legacy/
│   │   └── students.js
│   ├── mysql2/
│   │   ├── groups.js
│   │   └── students.js
│   └── sequelize/
│       ├── groups.js
│       └── students.js
├── sql/
│   ├── demo-queries.sql
│   └── schema.sql
├── .env
├── .env.example
├── package.json
├── package-lock.json
├── README.md
└── server.js
```

У цій структурі каталог `config/` відповідає за підключення до бази даних, `models/` містить Sequelize-моделі та опис зв’язків, `routes/` — маршрути API, а `sql/` — SQL-скрипти для створення схеми та демонстраційних запитів. Каталог `routes/legacy/` збережено з попередньої лабораторної роботи як окрему реалізацію з in-memory зберіганням даних, тому він відокремлений від маршрутів `mysql2` і `sequelize`, що працюють із MySQL. Файл `server.js` містить запуск Express-сервера та підключення маршрутів, а `README.md` — короткий опис проєкту та інструкцію із запуску.

## 3. README та запуск проєкту

У репозиторії серверної частини створено файл `README.md`, який містить короткий опис проєкту, перелік технологій, структуру бази даних, маршрути API та інструкцію щодо запуску.

### 3.1 Технології

У проєкті використано такі технології:
- `Node.js`
- `Express.js`
- `MySQL`
- `mysql2`
- `Sequelize`
- `dotenv`

### 3.2 Запуск проєкту

Для запуску серверної частини необхідно встановити залежності, налаштувати файл `.env`, виконати `sql/schema.sql` для створення бази даних і запустити сервер командою. Перед запуском потрібно коректно заповнити параметри підключення до MySQL у файлі `.env`:

```bash
npm start
```

Після запуску сервер буде доступний за адресою `http://localhost:3000`.
