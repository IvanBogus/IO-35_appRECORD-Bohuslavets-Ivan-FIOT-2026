## 2. Реалізація адаптивного інтерфейсу

### 2.1 Призначення frontend-частини

Frontend-частина проєкту реалізована як багатосторінковий вебзастосунок без використання фреймворків. Її призначення полягає у наданні користувачеві простого інтерфейсу для перегляду списку студентів, перегляду навчальних груп, заповнення форми додавання студента та переходу між основними розділами системи. Дані у клієнтській частині зберігаються у вигляді локального JavaScript-масиву.

### 2.2 Реалізовані сторінки

У складі frontend-частини реалізовано такі сторінки:
- `index.html` — головна сторінка з коротким описом застосунку та переходами до основних розділів;
- `students.html` — сторінка перегляду таблиці студентів;
- `groups.html` — сторінка перегляду навчальних груп із кількістю студентів у кожній групі;
- `add-student.html` — сторінка з формою введення даних студента та базовою перевіркою полів;
- `profile.html` — інформаційна сторінка профілю користувача;
- `login.html` — сторінка-заглушка для майбутнього функціоналу входу;
- `about.html` — сторінка з коротким описом застосунку.

### 2.3 Навігація між сторінками

Навігація організована у верхній частині сторінок і містить посилання на головну сторінку, сторінки студентів, груп, додавання студента, профілю та входу. Для навігаційних елементів реалізовано підсвічування активної сторінки. На малих екранах меню переходить у компактний формат із кнопкою відкриття, що забезпечує зручне використання інтерфейсу на мобільних пристроях.

### 2.4 Пошук і фільтрація

На сторінці `students.html` реалізовано відображення таблиці студентів із такими засобами відбору даних:
- пошук за ім’ям студента;
- пошук за ідентифікатором `ID`;
- фільтрація за навчальною групою;
- фільтрація за статусом студента.

Результати відбору оновлюються на сторінці динамічно. У разі відсутності записів, що відповідають заданим критеріям, відображається окреме повідомлення. Сторінка `groups.html` також містить переходи до сторінки студентів із попередньо вибраною групою через параметр URL.

### 2.5 Адаптивне компонування

Адаптивність інтерфейсу реалізовано за допомогою окремих CSS-файлів стилів і медіазапитів. Для екранів із шириною до `768px` передбачено зміну структури навігації, відображення кнопки мобільного меню, а також перебудову сіток і форм в одноколонковий вигляд. Адаптивно змінюються блоки карток, секції з фільтрами, форма додавання студента та елементи профілю, що забезпечує коректне відображення сторінок на різних типах пристроїв.

## 7. Реалізація серверної частини (Node.js + Express)

### 7.1 Загальний опис

Серверна частина проєкту реалізована на платформі Node.js із використанням фреймворку Express.js. REST API призначене для базової роботи з даними студентів через HTTP-запити. Для обробки тіла запиту у форматі JSON використовується вбудований middleware `express.json()`.

Дані студентів зберігаються у пам’яті сервера у вигляді масиву об’єктів. Такий підхід відповідає навчальному характеру проєкту та забезпечує виконання базових операцій без використання бази даних.

---

#### 7.2 Налаштування середовища

Для реалізації серверної частини використано Node.js та бібліотеку Express.js. Після встановлення залежностей сервер запускається командою:

```bash
npm start
```

### 7.3 Реалізація серверу

Основний файл серверної частини — `server.js`. У ньому створено Express-застосунок, налаштовано локальний порт `3000`, підключено обробку JSON-запитів і визначено набір маршрутів для роботи з ресурсом `students`.

### 7.4 Реалізовані API-ендпоінти

У сервері реалізовано такі маршрути:

- `GET /` — тестовий маршрут для перевірки роботи сервера.
- `GET /students` — повертає список усіх студентів.
- `GET /students/:id` — повертає одного студента за переданим ідентифікатором.
- `POST /students` — додає нового студента до in-memory масиву після базової перевірки полів `id`, `name`, `group`.
- `PUT /students/:id` — оновлює поля `name` та/або `group` для студента з указаним `id`.
- `DELETE /students/:id` — видаляє студента з масиву за ідентифікатором.

Для частини маршрутів передбачено базову обробку помилок: повернення `404`, якщо запис не знайдено, `400` у разі неповних даних під час створення та `409` у разі спроби додати студента з уже наявним `id`.

### 7.5 Тестування API

Відповідно до наявного опису в `README.md`, REST API охоплює операції отримання списку студентів, отримання одного студента, додавання, оновлення та видалення записів. Сервер працює локально за адресою `http://localhost:3000`.

### 7.7 Висновки

У ході виконання другої частини лабораторної роботи було реалізовано серверну частину вебзастосунку з використанням Node.js та Express.
